# Overview

Large-scale data migration requires evaluating:

* Available bandwidth.
* Data volume.
* Migration timeline.
* Transfer cost.
* Long-term storage requirements.

Migration methods:

| Method                           | Best Use Case                                        |
| -------------------------------- | ---------------------------------------------------- |
| 🌐 Online Transfer               | Smaller datasets or existing high-speed connectivity |
| 🔌 Direct Connect / ExpressRoute | Continuous enterprise migration                      |
| 📦 Physical Transfer             | Large datasets where bandwidth is limited            |

# AWS Approach

# Online vs Offline Migration Analysis

## Bandwidth Calculation

### Scenario: 70TB over 100 Mbps Internet Connection

Formula:

```text
Data Size × 1024 × 8 / Bandwidth Mbps
```

Calculation:

```text
70 × 1024 × 8 / 100

= 5,734 hours

= ~239 days
```

Result:

```text
❌ Not viable
```

---

## Scenario: 70TB over 1 Gbps Direct Connect

Calculation:

```text
70 × 1024 × 8 / 1000

= 573 hours

= ~24 days
```

Result:

```text
Possible, but expensive
```

# Snowball Edge Migration Process

# Week 1: Device Preparation

Activities:

* Order Snowball Edge from AWS Console.
* AWS ships device to the data center.

Timeline:

```text
2–3 business days
```

Install:

* Snowball Edge Client.

Unlock device using:

* Manifest file.
* Unlock code from AWS Console.

---

# Week 1–2: Data Copy

Copy data:

```bash
aws s3 cp /data/source s3://snowball-bucket --recursive
```

Alternative:

* Use Snowball File Interface.
* Mount device using NFS.

Validation:

* Verify checksum.
* Confirm all files copied successfully.

After completion:

1. Power off device.
2. Package securely.
3. Ship back to AWS.

---

# Week 2–3: AWS Data Ingestion

AWS process:

* Receive Snowball device.
* Import data into Amazon S3.
* Generate ingestion report.

Validation:

* Verify all files exist.
* Compare checksums.
* Confirm migration completion.

# Post-Migration Best Practice

> 💡 **Tip**
>
> Always configure lifecycle policies after migration.

Reason:

Data access patterns change:

```text
First Week:
Frequently Accessed

After Months:
Rarely Accessed
```

Recommended:

* Automatically move old data to cheaper storage tiers.
* Reduce storage costs.
* Maintain required availability.
