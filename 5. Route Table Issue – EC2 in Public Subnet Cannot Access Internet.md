# Route Table Issue – EC2 in Public Subnet Cannot Access Internet

> **Scenario:** An Amazon EC2 instance is deployed in a **public subnet** and has a **Public IP address**, but it cannot access the internet.

---

# Overview

An EC2 instance in a public subnet requires more than just a Public IP address to access the internet. Multiple networking components must be correctly configured, including the route table, Internet Gateway (IGW), Network ACLs (NACLs), and Security Groups.

This guide provides a systematic troubleshooting approach for identifying and resolving internet connectivity issues in both **AWS** and **Microsoft Azure** environments.

> 📌 **Important**
>
> A subnet is considered **public** only when its associated route table contains a default route (`0.0.0.0/0`) that points to an **Internet Gateway (IGW)**.

---

# AWS Approach

## Routing Checklist

Verify the following components **in the order listed**:

| Step  | Component                        | Verification                                                                                  |
| ----- | -------------------------------- | --------------------------------------------------------------------------------------------- |
| **1** | 🛣️ Subnet Association           | Verify that the subnet is associated with the correct route table.                            |
| **2** | 🌍 Route to Internet Gateway     | Confirm the route table contains **0.0.0.0/0 → igw-xxxxxxxx**.                                |
| **3** | 🌐 Internet Gateway              | Verify that an Internet Gateway exists and is attached to the VPC.                            |
| **4** | 🔗 Internet Gateway State        | Ensure the Internet Gateway status is **Attached**, not **Detached**.                         |
| **5** | 📍 Public IP / Elastic IP        | Verify the EC2 instance has a Public IP address or an Elastic IP assigned.                    |
| **6** | 🚧 Network ACL (NACL)            | Ensure outbound **TCP 80/443** and inbound ephemeral ports (**1024–65535**) are allowed.      |
| **7** | 🔒 Security Group Outbound Rules | Verify outbound traffic is permitted. By default, Security Groups allow all outbound traffic. |

---

## Check Route Table Associations

```bash id="u5g7re"
aws ec2 describe-route-tables \
--filters Name=association.subnet-id,Values=subnet-PUBLIC123
```

---

## Verify the Internet Gateway Attachment

```bash id="y9mwk2"
aws ec2 describe-internet-gateways \
--filters Name=attachment.vpc-id,Values=vpc-abc123
```

---

## Add a Missing Route to the Internet Gateway

```bash id="r2nq1b"
aws ec2 create-route \
--route-table-id rtb-PUBLIC789 \
--destination-cidr-block 0.0.0.0/0 \
--gateway-id igw-xxxxxxxx
```

# Root Cause Analysis (RCA)

## Root Cause 1

> ❌ The route table does **not** contain the required default route:

```text id="3j4yfa"
0.0.0.0/0 → Internet Gateway (IGW)
```

Without this route, internet-bound traffic cannot leave the VPC.

---

## Root Cause 2

> ❌ The Internet Gateway has been **detached** from the VPC.

An Internet Gateway may still exist within the AWS account but no longer be attached to the correct VPC.

---

## Root Cause 3

> ❌ The subnet is associated with the **wrong route table**.

For example:

* Public subnet associated with a **Private Route Table**
* Route points to a **NAT Gateway** instead of an **Internet Gateway**

---

## Root Cause 4

> ❌ The Network ACL blocks outbound traffic or inbound ephemeral return ports (**1024–65535**).

Even when the route table and Security Group are configured correctly, restrictive NACL rules can prevent successful communication.

---

## Resolution

* ✅ Attach the Internet Gateway to the VPC.
* ✅ Ensure the public route table contains:

```text id="qw4n6m"
0.0.0.0/0 → Internet Gateway (IGW)
```

* ✅ Associate the correct route table with the public subnet.
* ✅ Verify Network ACL rules allow outbound web traffic and inbound ephemeral response ports.
* ✅ Confirm the EC2 instance has a Public IP or Elastic IP.

# Azure Approach

Perform the following checks when troubleshooting outbound internet connectivity in Azure.

## Verification Checklist

### 🛣️ User-Defined Route (UDR)

Verify that the subnet's route table contains:

```text id="a8k5tp"
0.0.0.0/0 → Internet
```

Ensure the default route is **not** incorrectly directed to:

* Virtual Network (VNet)
* Network Virtual Appliance (NVA)
* Azure Firewall (unless intentionally configured)

---

### 🔥 Azure Firewall

If Azure Firewall is deployed:

* Verify outbound traffic is not being intercepted without the required:

  * Source Network Address Translation (SNAT)
  * Destination Network Address Translation (DNAT)
  * Firewall rules

---

### 🔗 Route Table Association

Confirm that the subnet is associated with the intended **User-Defined Route (UDR)** rather than relying on or conflicting with default system routes.

# Best Practice

> 💡 **Tip**
>
> Think of internet connectivity as a sequence of networking layers.

```text id="m2p8zd"
EC2
 │
 ▼
Security Group
 │
 ▼
Network ACL
 │
 ▼
Route Table
 │
 ▼
Internet Gateway
 │
 ▼
Internet
```

> 📌 **Important**
>
> Every layer must be correctly configured. A single misconfiguration can prevent internet connectivity, even if all other components are functioning properly.
