# Security – Restrict EC2 Access to Specific IP Addresses
 
> **Scenario:** A company wants only specific IP addresses, such as office networks or VPN ranges, to access Amazon EC2 instances.

---

# Overview

Allowing unrestricted access to cloud resources increases security risk. Production environments should restrict access using approved IP ranges and identity-based controls.

Common examples:

* 🔐 SSH access only from corporate office IPs.
* 🌐 Application access only from approved client networks.
* 🛡️ Administrative access only through VPN or controlled access paths.

The recommended security approach is to use multiple layers:

```text
Allowed IP Range
        │
        ▼
Security Group
        │
        ▼
Network ACL
        │
        ▼
Operating System Firewall
        │
        ▼
Application Security Controls
```

> 📌 **Important**
>
> Avoid using `0.0.0.0/0` for administrative ports such as SSH (`22`) in production environments.

---

# AWS Approach

## Primary Tool: Security Groups (Stateful Firewall)

Security Groups control inbound and outbound traffic at the instance level.

Recommended configuration:

* ✅ Add inbound rules using specific IP CIDR ranges.
* ✅ Avoid unrestricted access using `0.0.0.0/0`.
* ✅ Allow SSH only from approved office or VPN IP addresses.
* ✅ Allow application ports only from trusted client networks.
* ✅ Use Network ACLs as an additional subnet-level security layer.

---

# Security Group Configuration

## Allow SSH Only from Office IP

Replace the example IP with the approved office IP.

```bash
aws ec2 authorize-security-group-ingress \
--group-id sg-0abc123 \
--protocol tcp \
--port 22 \
--cidr 203.0.113.10/32
```

Example:

```text
203.0.113.10/32 = Single office public IP address
```

---

## Allow Application Port from Corporate Network

Example: Allow application traffic on port `8080`.

```bash
aws ec2 authorize-security-group-ingress \
--group-id sg-0abc123 \
--protocol tcp \
--port 8080 \
--cidr 203.0.113.0/24
```

Example:

```text
203.0.113.0/24 = Corporate network CIDR range
```

---

# Network ACL Additional Protection

Network ACLs provide subnet-level traffic filtering.

Example: Deny all other SSH attempts:

```bash
aws ec2 create-network-acl-entry \
--network-acl-id acl-abc123 \
--rule-number 50 \
--protocol tcp \
--rule-action deny \
--ingress \
--cidr-block 0.0.0.0/0 \
--port-range From=22,To=22
```

---

# Verify Security Group Rules

```bash
aws ec2 describe-security-groups \
--group-ids sg-0abc123
```

# Advanced Security: AWS WAF + Application Load Balancer

For application-level IP filtering:

Workflow:

```text
Allowed IP Addresses
        │
        ▼
AWS WAF IP Set
        │
        ▼
WAF Rule
        │
        ▼
Application Load Balancer
        │
        ▼
```

Benefits:

* WAF filters HTTP/HTTPS requests.
* Security Groups filter TCP-level traffic.
* Combined protection provides defense in depth.

# Root Cause Analysis (RCA)

## Risk Without IP Restriction

> ❌ Without IP filtering, any internet user can attempt connections to the EC2 instance.

Potential risks:

* Unauthorized access attempts.
* Automated scanning.
* Password brute-force attacks.
* Increased attack surface.

---

## Common Security Group Misconfiguration

> ❌ Allowing `0.0.0.0/0` on SSH port `22`.

Impact:

* Continuous brute-force attempts.
* Unauthorized login attempts visible in system authentication logs.
* Increased security monitoring workload.

---

# Resolution

Implement:

* ✅ Security Groups with approved IP ranges.
* ✅ Restricted SSH access through office/VPN IPs.
* ✅ AWS GuardDuty monitoring for suspicious activity.
* ✅ Regular review of access rules.

# Azure Approach

Azure provides similar network security controls.

---

## Network Security Group (NSG)

Configure inbound rules:

* Allow specific source IP addresses.
* Deny all other traffic using a lower-priority deny rule.

Example:

```text
Allowed Office IP
        │
        ▼
NSG Allow Rule
        │
        ▼
Application / VM
```

---

## Azure Firewall

Use Azure Firewall for centralized network security.

Benefits:

* Central IP filtering.
* Network-wide security policies.
* Logging and monitoring.
* Traffic inspection.

---

## Conditional Access (Microsoft Entra ID)

Use Conditional Access policies to:

* Restrict administrative portal access.
* Require approved IP ranges.
* Enforce identity-based security controls.

---

## Just-In-Time VM Access

JIT access opens SSH only when required.

Benefits:

* No permanently open SSH ports.
* Temporary access windows.
* Controlled source IP addresses.

---

# Azure JIT Access Example

```bash
az security jit-policy create \
--resource-group prod-rg \
--location eastus \
--vm-name prod-vm-01 \
--ports '[{"number":22,"protocol":"TCP","allowedSourceAddressPrefixes":["203.0.113.10"],"maxRequestAccessDuration":"PT3H"}]'
```

# Security Best Practices

> 💡 **Tip**
>
> Document all approved IP ranges and ownership information.

Maintain:

| Item              | Description                               |
| ----------------- | ----------------------------------------- |
| 📍 IP Range       | Approved office, VPN, or customer network |
| 👤 Owner          | Team responsible for the IP range         |
| 📅 Review Date    | Periodic security review schedule         |
| 🔄 Update Process | Procedure for IP changes                  |

---

# Operational Considerations

IP addresses can change due to:

* VPN changes.
* ISP rotation.
* Office network updates.
* Cloud network changes.

Maintain a process to update Security Groups and NSGs without accidentally locking out administrators.
