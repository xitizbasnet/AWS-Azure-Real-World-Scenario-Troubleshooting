# VPC – Private Server Internet Access (Without Making Public)

> **Applies to:** AWS VPC, Amazon EC2, NAT Gateway, Microsoft Azure Virtual Network (VNet)

> **Scenario:** An Amazon EC2 instance located in a **private subnet** needs outbound internet access (for example, to download packages or updates) **without being exposed to the public internet**.

---

# Overview

Instances deployed in a **private subnet** do not have direct internet connectivity. To enable **outbound-only** internet access while keeping the instance private, configure a **NAT Gateway**.

This guide explains how to implement this architecture in **AWS** and the equivalent approach in **Microsoft Azure**.

> 📌 **Important**
>
> A NAT Gateway provides **outbound internet connectivity only**. It does **not** allow inbound connections from the internet to private instances.

---

# AWS Approach

## Solution: NAT Gateway

Configure the following components in order:

| Step  | Component               | Configuration                                                               |
| ----- | ----------------------- | --------------------------------------------------------------------------- |
| **1** | 🌐 NAT Gateway          | Create a **NAT Gateway** in a **Public Subnet** (not a private subnet).     |
| **2** | 📍 Elastic IP           | Allocate an **Elastic IP (EIP)** and associate it with the NAT Gateway.     |
| **3** | 🛣️ Private Route Table | Update the private subnet route table with **0.0.0.0/0 → nat-xxxxxxxx**.    |
| **4** | 🌍 Public Route Table   | Ensure the public subnet route table contains **0.0.0.0/0 → igw-xxxxxxxx**. |

---

## Traffic Flow

```text
Private EC2
     │
     ▼
NAT Gateway (Public Subnet)
     │
     ▼
Internet Gateway (IGW)
     │
     ▼
Internet

Return Traffic

Internet
     │
     ▼
Internet Gateway (IGW)
     │
     ▼
NAT Gateway
     │
     ▼
Private EC2
```

> ℹ️ **Note**
>
> The NAT Gateway performs **Network Address Translation (NAT)**, allowing outbound traffic while preventing unsolicited inbound connections.

---

## Step 1 – Create a NAT Gateway

```bash id="t9y3ap"
aws ec2 create-nat-gateway \
--subnet-id subnet-PUBLIC123 \
--allocation-id eipalloc-xxxxxxxx
```

---

## Step 2 – Update the Private Subnet Route Table

```bash id="1hndsw"
aws ec2 create-route \
--route-table-id rtb-PRIVATE456 \
--destination-cidr-block 0.0.0.0/0 \
--nat-gateway-id nat-xxxxxxxx
```

---

## Verify Internet Connectivity

From the private EC2 instance (using AWS Systems Manager Session Manager):

```bash id="z08n2u"
curl -I https://google.com
```

Expected result:

* ✅ Internet connectivity is available.

Check the public IP address:

```bash id="e4glr1"
curl ifconfig.me
```

Expected result:

* The returned IP address should be the **NAT Gateway's Elastic IP**, **not** the EC2 instance's private IP address.

# Root Cause Analysis (RCA)

## Most Common Root Cause

> ❌ Without a NAT Gateway, a private EC2 instance has **no route to the internet**.

Common symptoms include:

* Package installation failures
* Operating system update failures
* `No route to host` errors
* Unable to reach external repositories

---

## Common Configuration Mistake

> ❌ Creating the NAT Gateway inside a **Private Subnet**.

A NAT Gateway **must** reside in a **Public Subnet** that has access to an Internet Gateway.

---

## Resolution

* ✅ Create the NAT Gateway in a **Public Subnet**.
* ✅ Associate an **Elastic IP** with the NAT Gateway.
* ✅ Update the **Private Subnet Route Table**:

```text
0.0.0.0/0 → NAT Gateway
```

# Azure Approach

Use one of the following methods to provide outbound internet access for private virtual machines.

## Option 1 – Azure NAT Gateway (Recommended)

* 🌐 Create an **Azure NAT Gateway**.
* 📍 Associate a **Public IP** or **Public IP Prefix**.
* 🔗 Associate the NAT Gateway with the **Private Subnet**.

---

## Option 2 – Azure Firewall

* Route outbound traffic through **Azure Firewall**.
* Provides:

  * Advanced filtering
  * Logging
  * Centralized security policies

> ⚠️ **Note**
>
> Azure Firewall provides more control but typically incurs higher costs than Azure NAT Gateway.

---

## Option 3 – Network Virtual Appliance (NVA)

Configure a **User-Defined Route (UDR)** to direct outbound traffic through a **Network Virtual Appliance (NVA)**.

---

## Create an Azure NAT Gateway

```bash id="w7m5co"
az network nat gateway create \
--name prod-nat-gw \
--resource-group prod-rg \
--location eastus
```

Associate the NAT Gateway with the private subnet:

```bash id="b2kj0x"
az network vnet subnet update \
--name private-subnet \
--vnet-name prod-vnet \
--resource-group prod-rg \
--nat-gateway prod-nat-gw
```

# Best Practice

> 💡 **Tip**
>
> Keep application servers in **Private Subnets** and provide outbound internet access through a **NAT Gateway** rather than assigning Public IP addresses.

### Cost Consideration

* AWS NAT Gateway:

  * Approximately **$0.045/hour**
  * Approximately **$0.045 per GB** of processed data

For development or low-cost environments:

* Consider using a **NAT Instance** (for example, **t3.nano**, approximately **$3/month**) instead of a managed NAT Gateway.

> 📌 **Recommendation**
>
> Use **NAT Gateway** for production workloads because it is highly available, managed by AWS, and requires minimal operational maintenance.
