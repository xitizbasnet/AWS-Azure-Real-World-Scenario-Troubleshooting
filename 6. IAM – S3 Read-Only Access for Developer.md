# IAM – S3 Read-Only Access for Developer

> **Scenario:** A developer requires **read-only access** to an Amazon S3 bucket and **must not** be able to modify or delete objects.

---

# Overview

Granting read-only access to storage resources should follow the **Principle of Least Privilege**, ensuring users receive only the permissions necessary to perform their tasks.

This guide demonstrates recommended approaches for providing **read-only access** to Amazon S3 in AWS and the equivalent implementation in Microsoft Azure.

> 📌 **Important**
>
> Avoid assigning broad permissions such as **AdministratorAccess** or **AmazonS3FullAccess** when only read access is required.

---

# AWS Approach

## Implementation Options

Choose one of the following approaches based on your organization's security requirements.

| Option | Method                                                                                                   | Recommendation                                                                 |
| ------ | -------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------ |
| **1**  | Attach the AWS managed policy **AmazonS3ReadOnlyAccess** to an IAM user or role.                         | Suitable for general read-only access.                                         |
| **2**  | Create a custom inline IAM policy for a specific S3 bucket.                                              | ✅ **Recommended** (Least Privilege).                                           |
| **3**  | Use **AWS IAM Identity Center (SSO)** with a permission set that grants S3 read-only access.             | Recommended for enterprise environments using centralized identity management. |
| **4**  | Add explicit **Deny** statements for `s3:DeleteObject` and `s3:PutObject` as a defense-in-depth measure. | Recommended for additional protection.                                         |

---

## Custom Least-Privilege Policy (Specific Bucket)

```json id="3txg1e"
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:GetObjectVersion",
        "s3:ListBucket",
        "s3:ListBucketVersions",
        "s3:GetBucketLocation"
      ],
      "Resource": [
        "arn:aws:s3:::company-prod-bucket",
        "arn:aws:s3:::company-prod-bucket/*"
      ]
    },
    {
      "Effect": "Deny",
      "Action": [
        "s3:DeleteObject",
        "s3:PutObject",
        "s3:PutBucketPolicy"
      ],
      "Resource": "*"
    }
  ]
}
```

---

## Attach the Policy to an IAM User

```bash id="j4wq7r"
aws iam put-user-policy \
--user-name dev-john \
--policy-name s3-read-only-prod \
--policy-document file://s3-read-policy.json
```

---

## Validate Permissions Using the IAM Policy Simulator

```bash id="a8yn2k"
aws iam simulate-principal-policy \
--policy-source-arn arn:aws:iam::123456:user/dev-john \
--action-names s3:DeleteObject \
--resource-arns arn:aws:s3:::company-prod-bucket/*
```

> ✅ **Expected Result**
>
> The simulation should confirm that the user **cannot** perform `s3:DeleteObject` on the specified bucket.

# Root Cause Analysis (RCA)

## Risk 1

> ❌ Assigning **AmazonS3FullAccess** or **AdministratorAccess** "temporarily" and forgetting to remove it.

This violates the Principle of Least Privilege and increases the risk of accidental or unauthorized changes.

---

## Risk 2

> ❌ Granting access to **all S3 buckets** instead of restricting access to the required bucket.

This exposes data beyond the user's operational requirements.

---

## Resolution

* ✅ Scope permissions to the specific bucket ARN.
* ✅ Grant only the required read actions.
* ✅ Use explicit **Deny** statements where appropriate.
* ✅ Perform regular permission reviews using **AWS IAM Access Analyzer**.

> 📌 **Recommendation**
>
> Audit IAM permissions on a quarterly basis to identify overly permissive access.

# Azure Approach

Use Azure Role-Based Access Control (RBAC) to grant read-only access to Azure Storage.

## Recommended Options

### 🔐 Azure RBAC

Assign the **Storage Blob Data Reader** role to the user.

Recommendations:

* Scope the assignment to the **specific Storage Account**.
* Prefer container-level scope where applicable.
* Avoid assigning permissions at the Subscription or Resource Group level unless required.

---

### ⏳ Shared Access Signature (SAS)

For temporary access:

* Generate a **read-only SAS token**.
* Configure an expiration time.
* Share the SAS token securely.

This approach is suitable for temporary external access without assigning permanent RBAC permissions.

---

## Assign the Storage Blob Data Reader Role

```bash id="n6pc9v"
az role assignment create \
--assignee dev-john@company.com \
--role "Storage Blob Data Reader" \
--scope /subscriptions/<SUB>/resourceGroups/prod-rg/providers/Microsoft.Storage/storageAccounts/prodstore
```

# Best Practice

> 💡 **Tip**
>
> Use **just-in-time (JIT)** or **time-bound** access whenever possible instead of granting permanent permissions.

### AWS

* AWS IAM Access Analyzer
* AWS IAM Identity Center (SSO)
* Least Privilege IAM Policies

### Azure

* Azure Privileged Identity Management (PIM)
* Azure Role-Based Access Control (RBAC)
* Time-limited role assignments

> 📌 **Security Recommendation**
>
> Restrict access to the **smallest possible scope** (specific bucket, container, or storage account) and review permissions regularly.
