# IAM Role for EC2 – Upload Logs to S3 Without Storing Credentials

> **Scenario:** An application running on an Amazon EC2 instance must upload log files to Amazon S3 **without storing AWS access keys or secret keys on the instance**.

---

# Overview

Applications running on Amazon EC2 should **never** store long-term AWS credentials (Access Key ID and Secret Access Key) in configuration files, environment variables, or source code.

The recommended approach is to attach an **IAM Role** to the EC2 instance through an **Instance Profile**. The AWS SDK automatically retrieves temporary credentials from the **EC2 Instance Metadata Service (IMDS)**, eliminating the need to manage credentials manually.

> 📌 **Important**
>
> IAM Roles provide **temporary credentials** that are automatically rotated and securely delivered to the EC2 instance through the Instance Metadata Service (`169.254.169.254`).

---

# AWS Approach

## Solution: IAM Instance Profile (IAM Role Attached to EC2)

Configure the following components in order:

| Step  | Component                        | Configuration                                                                                                                                                              |
| ----- | -------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **1** | 🔐 Create IAM Role               | Configure the trusted entity as the **EC2 service**.                                                                                                                       |
| **2** | 📦 Attach IAM Policy             | Grant only the required permissions, such as `s3:PutObject` for the target log bucket.                                                                                     |
| **3** | 🖥️ Create Instance Profile      | Attach the IAM Role to an Instance Profile and associate it with the EC2 instance.                                                                                         |
| **4** | ⚙️ AWS SDK Authentication        | The AWS SDK automatically retrieves temporary credentials from the EC2 Instance Metadata Service (`169.254.169.254`). No credentials are required in the application code. |
| **5** | 🔄 Automatic Credential Rotation | Temporary credentials rotate automatically approximately every **6 hours**, requiring no manual rotation.                                                                  |

---

## Step 1 – Create an IAM Role

```bash id="j4r7nt"
aws iam create-role \
--role-name ec2-log-uploader \
--assume-role-policy-document '{
  "Version":"2012-10-17",
  "Statement":[{
    "Effect":"Allow",
    "Principal":{"Service":"ec2.amazonaws.com"},
    "Action":"sts:AssumeRole"
  }]
}'
```

---

## Step 2 – Attach an S3 Write Policy

```bash id="m8x2fd"
aws iam put-role-policy \
--role-name ec2-log-uploader \
--policy-name s3-logs-write \
--policy-document '{
  "Statement":[{
    "Effect":"Allow",
    "Action":[
      "s3:PutObject",
      "s3:PutObjectAcl"
    ],
    "Resource":"arn:aws:s3:::prod-app-logs/*"
  }]
}'
```

---

## Step 3 – Create an Instance Profile and Attach the IAM Role

```bash id="r2w6kp"
aws iam create-instance-profile \
--instance-profile-name ec2-log-uploader
```

```bash id="v5n8yb"
aws iam add-role-to-instance-profile \
--instance-profile-name ec2-log-uploader \
--role-name ec2-log-uploader
```

---

## Step 4 – Associate the Instance Profile with the EC2 Instance

```bash id="q7c1le"
aws ec2 associate-iam-instance-profile \
--instance-id i-0abc123 \
--iam-instance-profile Name=ec2-log-uploader
```

---

## Verify from the EC2 Instance

No AWS credentials are required in the application or command line.

```bash id="x4m9zs"
aws s3 cp /var/log/app.log s3://prod-app-logs/
```

> ✅ **Expected Result**
>
> The upload succeeds because the AWS CLI automatically retrieves temporary credentials from the attached IAM Role.

# Root Cause Analysis (RCA)

## Common Anti-Pattern

> ❌ Hardcoding `AWS_ACCESS_KEY_ID` and `AWS_SECRET_ACCESS_KEY` in:

* Application source code
* Configuration files
* `.env` files
* Shell scripts

---

## Security Risk

> ❌ If the EC2 instance is compromised, attackers gain access to **long-term credentials**, which can be used indefinitely until manually revoked.

IAM Roles eliminate this risk by issuing **temporary credentials** that expire automatically.

---

## Resolution

* ✅ Attach an IAM Role to the EC2 instance using an Instance Profile.
* ✅ Grant only the minimum required permissions (for example, `s3:PutObject`).
* ✅ Allow the AWS SDK to obtain credentials automatically.
* ✅ Use **AWS CloudTrail** to audit API activity performed by the IAM Role.

> 📌 **Recommendation**
>
> Never distribute or store long-term AWS credentials on EC2 instances.

# Azure Approach

Microsoft Azure provides **Managed Identity**, which is the equivalent of an AWS IAM Instance Profile.

## Managed Identity Options

### 🖥️ System-Assigned Managed Identity

* Automatically created with the virtual machine.
* Lifecycle is tied to the VM.
* Automatically deleted when the VM is deleted.

---

### 👥 User-Assigned Managed Identity

* Independent Azure resource.
* Can be assigned to multiple virtual machines.
* Suitable for shared application identities.

---

## Required Permission

Assign the following Azure RBAC role:

* **Storage Blob Data Contributor**

Scope the assignment to the required Storage Account whenever possible.

---

## Enable a System-Assigned Managed Identity

```bash id="t6k2va"
az vm identity assign \
--name prod-vm-01 \
--resource-group prod-rg
```

---

## Assign the Required Role

```bash id="f9u4xn"
az role assignment create \
--assignee <managed-identity-object-id> \
--role "Storage Blob Data Contributor" \
--scope /subscriptions/<SUB>/resourceGroups/prod-rg/providers/Microsoft.Storage/storageAccounts/prodlogs
```

# Best Practice

> 💡 **Tip**
>
> Use **identity-based authentication** instead of storing credentials.

### AWS

* IAM Role
* Instance Profile
* Temporary STS Credentials
* EC2 Instance Metadata Service (IMDS)
* AWS CloudTrail Auditing

### Azure

* Managed Identity
* Azure RBAC
* Microsoft Entra ID Authentication
* Azure Activity Logs
