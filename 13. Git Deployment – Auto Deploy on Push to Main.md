# Git Deployment – Auto Deploy on Push to Main

> **Scenario:** An application should automatically deploy whenever code is pushed to the **main branch**.

---

# Overview

Continuous Integration and Continuous Deployment (CI/CD) automates the process of:

* Building application artifacts.
* Running validation checks.
* Creating container images.
* Publishing images to registries.
* Deploying applications automatically.
* Maintaining deployment history and audit trails.

A properly configured CI/CD pipeline eliminates manual deployment processes such as:

* SSH-based deployments.
* Manual Docker builds.
* Manual server updates.
* Human errors during releases.

> 📌 **Important**
>
> A production deployment pipeline should provide automation, security, traceability, and rollback capability.

---

# AWS Approach

AWS provides multiple approaches for automated deployments.

## Option 1: GitHub Actions (Recommended for GitHub Repositories)

GitHub Actions can automatically trigger deployments when code is pushed to the `main` branch.

---

# GitHub Actions Workflow Example

File:

```text
.github/workflows/deploy-to-prod.yml
```

```yaml
name: Deploy to Production

on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest

    permissions:
      id-token: write
      contents: read

    steps:

    - uses: actions/checkout@v4

    - name: Configure AWS (OIDC - no static keys)
      uses: aws-actions/configure-aws-credentials@v4
      with:
        role-to-assume: arn:aws:iam::123456:role/github-actions-role
        aws-region: ap-south-1

    - name: Build & Push Docker Image
      run: |
        aws ecr get-login-password | docker login --username AWS \
        --password-stdin 123456.dkr.ecr.ap-south-1.amazonaws.com

        docker build -t prod-api:${{ github.sha }} .

        docker push 123456.dkr.ecr.ap-south-1.amazonaws.com/prod-api:${{ github.sha }}

    - name: Deploy to ECS
      run: |
        aws ecs update-service \
        --cluster prod \
        --service api-service \
        --task-definition api-task:latest \
        --force-new-deployment
```

# Security Best Practice: AWS OIDC Authentication

Traditional approach:

```text
❌ Store AWS Access Keys in GitHub Secrets
```

Recommended approach:

```text
✅ GitHub OIDC
        │
        ▼
Assume IAM Role
        │
        ▼
Temporary AWS Credentials
```

Benefits:

* No static AWS credentials.
* Automatic credential expiration.
* Improved security auditing.
* Reduced credential leakage risk.

# Root Cause Analysis (RCA)

## Root Cause

> ❌ Manual deployments create inconsistency, missed steps, and increased human error risk.

Common problems:

* Different deployment commands used by different engineers.
* Missing configuration changes.
* No deployment audit trail.
* Difficult rollback process.
* Production access through SSH.

---

# Resolution

Implement an automated CI/CD pipeline:

```text
Git Push
    │
    ▼
Automated Build
    │
    ▼
Automated Testing
    │
    ▼
Container Image Push
    │
    ▼
Automated Deployment
    │
    ▼
Monitoring + Rollback
```

Benefits:

* ✅ Same deployment process every time.
* ✅ Complete audit history.
* ✅ Reduced manual errors.
* ✅ Faster rollback.
* ✅ No SSH access required for deployments.

# Azure Approach

Azure provides Azure Pipelines for automated deployments.

Supported sources:

* GitHub repositories.
* Azure Repos.

Deployment trigger:

```text
Push to main branch
        │
        ▼
Azure Pipeline Starts
```

---

# Azure Pipelines Example

File:

```text
azure-pipelines.yml
```

```yaml
trigger:
  branches:
    include:
      - main

stages:

- stage: Build

  jobs:

  - job: BuildAndPush

    steps:

    - task: Docker@2
      inputs:
        containerRegistry: 'prod-acr-connection'
        repository: 'prod-api'
        command: 'buildAndPush'
        tags: '$(Build.SourceVersion)'


- stage: Deploy

  jobs:

  - deployment: DeployToAKS

    environment: 'production'

    strategy:

      runOnce:

        deploy:

          steps:

          - task: KubernetesManifest@0
            inputs:
              action: 'deploy'
              manifests: 'k8s/deployment.yaml'
```

# Production Deployment Best Practices

## Approval Gates

> 💡 **Tip**
>
> Add a manual approval gate before production deployment.

Recommended workflow:

```text
Developer Push
       │
       ▼
Auto Deploy to Staging
       │
       ▼
Automated Tests
       │
       ▼
Manual Production Approval
       │
       ▼
Production Deployment
```

Benefits:

* Prevents accidental production releases.
* Provides change control.
* Allows final validation before customer impact.
