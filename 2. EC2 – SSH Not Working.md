# EC2 – SSH Not Working

> **Applies to:** AWS EC2, AWS Systems Manager (SSM), Microsoft Azure Virtual Machines

> **Audience:** Cloud Engineers, System Administrators, DevOps Engineers

> **Scenario:** An Amazon EC2 instance is **running**, but the SSH connection fails or times out.

---

# Overview

SSH connectivity failures are one of the most common issues encountered when managing Linux virtual machines. The problem can originate from network configuration, authentication, operating system services, or instance placement within the network.

This guide provides a structured troubleshooting workflow for both **AWS** and **Microsoft Azure** environments.

> 📌 **Important**
>
> Before troubleshooting SSH, verify that the EC2 instance is in the **Running** state and has passed all health checks.

---

# AWS Approach

## Possible Reasons and Fixes

Verify the following items **in the order listed**:

| Step  | Component                | Verification / Resolution                                                                                                                                     |
| ----- | ------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **1** | 🔒 Security Group        | Ensure **TCP port 22** is allowed from your public IP or **0.0.0.0/0** (temporary for testing).                                                               |
| **2** | 🔑 Key Pair              | Verify that you are using the correct **.pem** file associated with the EC2 instance.                                                                         |
| **3** | 👤 Username              | Use the correct SSH username based on the AMI.<br><br>• Amazon Linux = **ec2-user**<br>• Ubuntu = **ubuntu**<br>• RHEL = **ec2-user**<br>• Debian = **admin** |
| **4** | 📄 Key Permissions       | Ensure the **.pem** file permissions are set to **chmod 400**.                                                                                                |
| **5** | 🌐 Private Subnet        | If the instance resides in a private subnet, direct SSH is unavailable. Use a **Bastion Host** or **AWS Systems Manager Session Manager**.                    |
| **6** | ✅ Instance Status Checks | If instance status checks fail, perform a **Stop** and **Start** operation to migrate the instance to a healthy host.                                         |
| **7** | ⚙️ SSH Daemon            | Verify that the **sshd** service is running. Use the EC2 Serial Console or AWS Systems Manager if SSH is unavailable.                                         |

---

## Fix Key Permissions

```bash id="4lq2ew"
chmod 400 my-key.pem
```

---

## Connect Using SSH

Amazon Linux / RHEL

```bash id="6ph2gu"
ssh -i my-key.pem ec2-user@<public-ip>
```

Ubuntu

```bash id="r7mv6g"
ssh -i my-key.pem ubuntu@<public-ip>
```

---

## Connect Using AWS Systems Manager Session Manager

> 💡 **Tip**
>
> Session Manager provides secure shell access **without requiring an SSH key or opening port 22**.

```bash id="9dj4pw"
aws ssm start-session --target i-0abc123def456
```

---

## Verify the SSH Service

```bash id="y63b1z"
sudo systemctl status sshd
sudo systemctl start sshd
```

# Root Cause Analysis (RCA)

## Most Common Root Cause

> ❌ Port **22** is not allowed in the Security Group **OR**
>
> ❌ An incorrect SSH username is being used for the selected AMI.

---

## Second Most Common Root Cause

> ❌ The EC2 instance is deployed in a **private subnet**, preventing direct SSH access.

Use either:

* AWS Systems Manager Session Manager
* Bastion Host

---

## Resolution

* ✅ Open TCP port **22** in the Security Group (when appropriate).
* ✅ Verify the correct SSH username for the operating system.
* ✅ Confirm the correct **.pem** key pair is being used.
* ✅ Prefer **AWS Systems Manager Session Manager** as a long-term replacement for traditional SSH access.

---

# Azure Approach

Perform the following checks when troubleshooting SSH connectivity to an Azure Virtual Machine.

## Verification Checklist

* 🔒 **Network Security Group (NSG)**

  * Ensure an inbound rule exists allowing **TCP port 22**.

* 🖥️ **Azure Bastion**

  * Provides secure browser-based SSH.
  * No Public IP required.
  * No need to expose port **22**.

* ⏱️ **Just-in-Time (JIT) Access**

  * Use Microsoft Defender for Cloud to open port **22** only when required.

* 👤 **Username**

  * Azure Linux virtual machines use the **administrator username specified during VM creation**.

---

## Connect Using Azure Bastion

> 💡 **Tip**
>
> Azure Bastion enables secure SSH access without exposing the virtual machine to the public internet.

```bash id="v1fj0m"
az network bastion ssh \
--name prod-bastion \
--resource-group prod-rg \
--target-resource-id <vm-id> \
--auth-type ssh-key \
--username azureuser \
--ssh-key my-key.pem
```

# Best Practice

> 💡 **Tip**
>
> The recommended approach is to **eliminate direct SSH access whenever possible**.

### AWS

* ✅ AWS Systems Manager Session Manager
* No SSH key management
* No inbound Security Group rule for port 22
* Reduced attack surface

### Azure

* ✅ Azure Bastion
* Browser-based SSH
* No Public IP required
* No permanently open port 22

> 📌 **Security Recommendation**
>
> Using **AWS Systems Manager Session Manager** or **Azure Bastion** significantly improves security by removing the need to expose SSH over the internet.
