# Docker Container Starts but Exits Immediately

> **Scenario:** A Docker container starts successfully but exits after a few seconds with an exit code.

---

# Overview

Containers do not behave like traditional virtual machines. A container lifecycle is directly tied to the lifecycle of its **main process**.

If the primary process exits, the container stops automatically.

Common reasons include:

* Incorrect `CMD` or `ENTRYPOINT`
* Application startup failures
* Missing environment variables
* Incorrect file permissions
* Invalid startup scripts
* Memory-related termination

This guide provides a structured troubleshooting approach for identifying why containers exit immediately.

> 📌 **Important**
>
> Always check the container logs first. The logs usually provide the fastest indication of why the application stopped.

---

# AWS Approach

## Debugging Steps

Perform the following checks in order:

| Step  | Component                | Verification                                                                                                         |
| ----- | ------------------------ | -------------------------------------------------------------------------------------------------------------------- |
| **1** | 🚪 Exit Code             | Run `docker ps -a` and check the exit code shown in the STATUS column.                                               |
| **2** | 📄 Container Logs        | Review logs using `docker logs <container-id>`. This is the most important troubleshooting step.                     |
| **3** | 🐚 Interactive Debugging | Start the container interactively using `/bin/sh` to inspect the environment.                                        |
| **4** | ⚙️ CMD / ENTRYPOINT      | Verify that the Dockerfile startup command is correct and points to valid files.                                     |
| **5** | 🔑 Environment Variables | Confirm all required environment variables are available. Missing variables can cause immediate application failure. |

# Common Docker Exit Codes

| Exit Code | Meaning                | Troubleshooting Action                                                                             |
| --------- | ---------------------- | -------------------------------------------------------------------------------------------------- |
| **0**     | ✅ Clean exit           | The main command completed successfully. Verify whether the container is expected to keep running. |
| **1**     | ❌ Application error    | Review application logs and startup configuration.                                                 |
| **126**   | 🔒 Permission denied   | Check whether the ENTRYPOINT script has execute permissions.                                       |
| **127**   | ❌ Command not found    | Verify the CMD path or executable name.                                                            |
| **137**   | ⚠️ Out of Memory (OOM) | Increase memory allocation or investigate memory usage.                                            |
| **139**   | ❌ Segmentation fault   | Investigate application or native library crashes.                                                 |

---

# Most Common Causes

| Cause                           | Description                                                                 |
| ------------------------------- | --------------------------------------------------------------------------- |
| ❌ Incorrect CMD Path            | The CMD references a script or binary that does not exist inside the image. |
| ❌ Script Completes Immediately  | The startup script finishes execution, causing the container to stop.       |
| ❌ Missing Environment Variables | Required configuration values are unavailable during startup.               |
| ❌ Port Binding Failure          | The application cannot bind because the port is already in use.             |

# Root Cause Analysis (RCA)

## Root Cause

> ❌ The `CMD` or `ENTRYPOINT` process exits immediately, either intentionally or because of an application error.

Container behavior:

```text id="k6m2jd"
Container Lifecycle
        =
Main Process Lifecycle
```

When the main process exits, Docker stops the container.

---

## Example Issue

Incorrect startup script:

```bash id="p5w8hz"
CMD ["bash", "start.sh"]
```

Problem:

* The script runs.
* The script completes.
* No foreground process remains.
* Container exits.

---

## Resolution

Ensure the container starts a long-running foreground process.

Use `exec` in startup scripts:

```bash id="x2v7nc"
exec node app.js
```

Benefits:

* Replaces the shell process with the application process.
* Allows proper signal handling.
* Keeps the container running.

# Recommended Container Startup Pattern

```text id="z4n8qs"
Docker Container Starts
          │
          ▼
ENTRYPOINT / CMD
          │
          ▼
Application Runs in Foreground
          │
          ▼
Container Remains Running
```

# Best Practice

> 💡 **Tip**
>
> Containers are **not virtual machines**. They exit when their main process exits.

Follow these practices:

* ✅ Run applications in the foreground.
* ✅ Avoid background daemon processes inside containers.
* ✅ Use `exec` in startup scripts.
* ✅ Validate CMD and ENTRYPOINT instructions.
* ✅ Monitor container exit codes.

> 📌 **Operational Recommendation**
>
> Before deploying containers to ECS, AKS, or production environments, test the container locally using `docker run` and validate that the main process remains active.
