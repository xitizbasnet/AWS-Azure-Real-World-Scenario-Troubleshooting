# Production Slowness During Peak Traffic – Troubleshoot and Scale

> **Scenario:** A production EC2 application becomes very slow during peak traffic. The objective is to diagnose the bottleneck and scale the environment.

---

# Overview

Performance issues during peak traffic usually occur because one or more components cannot handle increased demand.

Common bottlenecks:

* 🖥️ EC2 CPU exhaustion.
* 🧠 Memory pressure or memory leaks.
* 💾 Database overload.
* 🌐 Network limitations.
* 📦 Missing caching strategy.
* 📈 Lack of automatic scaling.

The troubleshooting approach should always start with measurement before scaling.

```text
Traffic Increase
       │
       ▼
Monitor Metrics
       │
       ▼
Identify Bottleneck
       │
       ▼
Apply Targeted Fix
       │
       ▼
Scale and Validate
```

> 📌 **Important**
>
> Do not immediately increase server size without identifying the actual bottleneck. Scaling the wrong component increases cost without solving the issue.

# AWS Approach

# Troubleshooting Steps (Perform First)

Follow the troubleshooting sequence below.

---

## 1. Check Amazon CloudWatch Metrics

Review EC2 performance metrics:

Monitor:

* CPU utilization.
* Memory usage (using CloudWatch Agent).
* Network In/Out.
* Disk I/O.

Questions to answer:

* Is CPU constantly above capacity?
* Is memory exhausted?
* Is network throughput reaching limits?
* Is disk I/O causing delays?

---

## 2. Review Application Logs

Check CloudWatch Logs for:

* Slow database queries.
* Application errors.
* Request timeouts.
* Failed API calls.
* Exception stack traces.

---

## 3. Check Amazon RDS Performance

Database bottlenecks are common during traffic spikes.

Review:

* Database CPU utilization.
* IOPS.
* Active connections.
* Slow query logs.

---

## 4. Check Application Load Balancer Metrics

Review ALB metrics:

| Metric                      | Purpose                              |
| --------------------------- | ------------------------------------ |
| `TargetResponseTime`        | Measures backend response latency    |
| `HTTPCode_Target_5XX_Count` | Detects backend application failures |

---

## 5. Diagnose Directly on the Server

Run:

### Check CPU Usage

```bash
top -bn1
```

Shows CPU consumption by process.

---

### Check Memory Pressure

```bash
free -h
```

Identifies available and used memory.

---

### Check Disk I/O

```bash
iostat -x 1 3
```

Shows disk latency and I/O wait.

---

### Check Active Connections

```bash
netstat -an | grep ESTABLISHED | wc -l
```

Shows the number of active network connections.

---

### Check Disk Space

```bash
df -h
```

A full disk can cause:

* Slow writes.
* Application failures.
* Log generation issues.

# Scaling Solutions Based on Root Cause

Choose scaling actions based on the identified bottleneck.

| Bottleneck           | Solution                                                                 |
| -------------------- | ------------------------------------------------------------------------ |
| 🔥 CPU-bound         | Vertical scaling (upgrade instance type) OR horizontal scaling using ASG |
| 🧠 Memory-bound      | Increase RAM or fix memory leak                                          |
| 🗄️ Database-bound   | Add RDS Read Replica, route read traffic, enable ElastiCache             |
| 🌐 Network-bound     | Move to network-optimized instances (example: `t3` → `m5n`)              |
| 📈 Long-term scaling | Configure Auto Scaling Group with Target Tracking                        |

---

# Immediate Solution: Resize Instance

> ⚠️ This requires downtime.

Stop the instance:

```bash
aws ec2 stop-instances \
--instance-ids i-0abc123
```

Change instance type:

```bash
aws ec2 modify-instance-attribute \
--instance-id i-0abc123 \
--instance-type '{"Value":"t3.xlarge"}'
```

Start instance:

```bash
aws ec2 start-instances \
--instance-ids i-0abc123
```

---

# Recommended Solution: Auto Scaling Group

Auto Scaling provides:

* Automatic capacity adjustment.
* No manual intervention.
* Better availability.
* Improved handling of traffic spikes.

Example Target Tracking Scaling Policy:

```bash
aws autoscaling put-scaling-policy \
--auto-scaling-group-name prod-web-asg \
--policy-name scale-on-cpu \
--policy-type TargetTrackingScaling \
--target-tracking-configuration \
'{"TargetValue":60,"PredefinedMetricSpecification":{"PredefinedMetricType":"ASGAverageCPUUtilization"}}'
```

# Root Cause Analysis (RCA)

## Common Root Causes

### 1. No Auto Scaling

Problem:

> A single EC2 instance receives all traffic and becomes overloaded.

Solution:

* Deploy multiple instances.
* Use Auto Scaling Group.
* Place instances behind ALB.

---

### 2. N+1 Database Queries

Problem:

> Application executes unnecessary database queries during high traffic.

Symptoms:

* High database CPU.
* Slow API responses.
* Increased query latency.

Solution:

* Optimize queries.
* Add indexes.
* Use read replicas.

---

### 3. No Caching

Problem:

> Repeated expensive queries hit the database continuously.

Solution:

* Implement Redis caching.
* Use Amazon ElastiCache.
* Cache frequently accessed data.

---

### 4. Memory Leak

Problem:

> Application memory usage increases gradually until performance degrades.

Solution:

* Analyze memory usage.
* Fix application leak.
* Increase memory temporarily if required.

---

# RCA Resolution Summary

## Immediate Fix

* Resize instance.
* Add additional EC2 instances.
* Restart unhealthy components.

## Long-Term Fix

Implement:

* ✅ Auto Scaling Groups.
* ✅ ElastiCache.
* ✅ RDS Read Replicas.
* ✅ Load testing before expected traffic events.

# Azure SQL Read Replicas

Use read replicas for:

* Reporting workloads.

---

# Azure VMSS Autoscale Example

Create autoscale configuration:

```bash
az monitor autoscale create \
--resource-group prod-rg \
--name prod-autoscale \
--resource prod-vmss \
--resource-type Microsoft.Compute/virtualMachineScaleSets \
--min-count 2 \
--max-count 20 \
--count 3
```

---

Create CPU-based scale rule:

```bash
az monitor autoscale rule create \
--autoscale-name prod-autoscale \
--resource-group prod-rg \
--condition "Percentage CPU > 70 avg 5m" \
--scale out 2
```

> 💡 **Tip**

* Discover bottlenecks early.
