# ECS Deployment Issue – Task Keeps Restarting

> **Scenario:** A containerized application is deployed to **Amazon ECS**, but the task repeatedly enters the **STOPPED** state and is automatically restarted.

---

# Overview

When an Amazon ECS task continuously restarts, the underlying issue is typically related to the application, container image, task configuration, resource allocation, permissions, or health checks.

This guide provides a structured troubleshooting workflow to identify the root cause and resolve repeated task failures.

> 📌 **Important**
>
> Begin troubleshooting by examining the **stopped task logs** and the **container exit code**. These two items usually provide the fastest path to identifying the root cause.

---

# AWS Approach

## Debugging Checklist

Perform the following checks **in the order listed**:

| Step  | Component            | Verification                                                                                                         |
| ----- | -------------------- | -------------------------------------------------------------------------------------------------------------------- |
| **1** | 📄 Stopped Task Logs | Navigate to **ECS → Cluster → Service → Tasks → Stopped Tasks → Logs** and review the failure details.               |
| **2** | 📊 CloudWatch Logs   | Review the container log group (`/ecs/<service-name>`) for application errors.                                       |
| **3** | 🚪 Exit Code         | Determine why the container exited by reviewing the exit code.                                                       |
| **4** | 📦 Task Definition   | Verify the container image URI, environment variables, and referenced secrets.                                       |
| **5** | 🔐 IAM Permissions   | Ensure the **Task Execution Role** includes permissions such as `ecr:GetAuthorizationToken` and `ecr:BatchGetImage`. |
| **6** | 💾 CPU and Memory    | Compare the task definition resource limits with the application's actual requirements.                              |
| **7** | ❤️ Health Check      | Verify the Application Load Balancer (ALB) health check path and expected HTTP status code.                          |

---

## Check the Task Exit Code and Failure Reason

```bash id="k2t9mc"
aws ecs describe-tasks \
--cluster prod \
--tasks <task-id> \
--query 'tasks[0].containers[0].{exitCode:exitCode,reason:reason}'
```

---

## Stream CloudWatch Logs

```bash id="x7w4fh"
aws logs tail /ecs/prod-api \
--since 1h \
--follow
```

---

## Common Container Exit Codes

| Exit Code | Meaning                     | Recommended Action                                                                           |
| --------- | --------------------------- | -------------------------------------------------------------------------------------------- |
| **0**     | ✅ Clean exit                | Verify that the application is intended to terminate. Check the Docker `CMD` or entry point. |
| **1**     | ❌ Application error         | Review application logs for startup or runtime failures.                                     |
| **137**   | ⚠️ Out of Memory (OOM Kill) | Increase the task memory allocation or investigate memory leaks.                             |
| **139**   | ❌ Segmentation Fault        | Investigate native code or library crashes.                                                  |
| **143**   | ⚠️ SIGTERM                  | ECS stopped the task, often because the health check failed or a deployment occurred.        |

# Common Root Causes

| Issue                          | Description                                                                                                                               |
| ------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------- |
| ❌ Missing Environment Variable | Required variables (for example, `DATABASE_URL`) are missing, causing the application to fail during startup.                             |
| ❌ Incorrect Docker Image Tag   | ECS pulls an outdated or incorrect container image.                                                                                       |
| ❌ Insufficient Memory          | Memory allocation is too low, resulting in an **Exit Code 137 (OOM Kill)**.                                                               |
| ❌ Health Check Failure         | The configured health check endpoint returns a non-200 HTTP status, causing ECS to terminate and replace the task.                        |
| ❌ Secrets Access Failure       | The task cannot retrieve secrets from AWS Secrets Manager or AWS Systems Manager Parameter Store because of insufficient IAM permissions. |

# Root Cause Analysis (RCA)

## Exit Code 137 (Out of Memory)

> ❌ The container exceeded its allocated memory and was terminated by the operating system.

### Resolution

* Increase the task memory allocation.
* Investigate the application for memory leaks.

---

## Exit Code 1 (Application Startup Failure)

> ❌ The application exited during startup.

Common causes include:

* Missing `DATABASE_URL`
* Missing environment variables
* Invalid configuration
* Application initialization errors

### Resolution

* Verify environment variables.
* Validate task definition configuration.
* Review application logs.

---

## Exit Code 143 (SIGTERM Loop)

> ❌ ECS continuously stops and restarts the task because the configured health check fails.

Example:

* `/health` returns **HTTP 500**
* Database is unreachable
* Dependent services are unavailable

### Resolution

* Verify health check configuration.
* Confirm backend dependencies are available.
* Review ALB health check settings.

# Recommended Troubleshooting Workflow

```text id="p4r8yb"
Stopped Task
      │
      ▼
Read Container Logs
      │
      ▼
Identify Exit Code
      │
      ▼
Determine Root Cause
      │
      ▼
Update Task Definition or Application
      │
      ▼
Redeploy Service
```

> 📌 **Recommendation**
>
> Always review the **stopped task logs first**, then use the **exit code** to guide further troubleshooting.

# Azure Approach

The equivalent troubleshooting process depends on the Azure container service in use.

## Azure Container Instances (ACI)

Review container logs using the Azure Portal or Azure CLI.

```bash id="m5v3jq"
az container logs \
--resource-group prod-rg \
--name prod-api-container
```

---

## Azure Kubernetes Service (AKS)

Retrieve logs from the previous container instance after a crash.

```bash id="d8y2nw"
kubectl logs <pod-name> \
--previous \
-n prod
```

Review Kubernetes events.

```bash id="q1h7es"
kubectl describe pod <pod-name> \
-n prod | grep -A 10 Events
```

---

## Azure Container Apps

Review **Revision Logs** using the associated **Log Analytics Workspace** to identify startup failures, health check issues, or application exceptions.

# Best Practice

> 💡 **Tip**
>
> Configure container restart policies with an appropriate **backoff strategy**.

Benefits include:

* Prevents thousands of rapid restart attempts.
* Reduces unnecessary image pulls from Amazon ECR.
* Lowers Amazon CloudWatch Logs ingestion costs.
* Improves overall service stability during deployment failures.

> 📌 **Operational Recommendation**
>
> Monitor:
>
> * Container exit codes
> * Memory utilization
> * CPU utilization
> * Health check success rate
> * CloudWatch Logs
>
> These metrics typically identify the root cause of recurring task failures before they affect production workloads.
