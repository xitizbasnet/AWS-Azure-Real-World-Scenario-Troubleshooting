# EC2 – Application Not Accessible on Port 8080

> **Applies to:** AWS EC2, Amazon VPC, Microsoft Azure Virtual Machines

> **Audience:** Cloud Engineers, System Administrators, DevOps Engineers

> **Scenario:** An Amazon EC2 instance has been launched, and an application is deployed on **port 8080**, but users cannot access the application from the internet.

---

# Overview

When an application running on an EC2 instance is inaccessible over **port 8080**, multiple networking and operating system layers should be verified. This guide provides a structured troubleshooting approach for both **AWS** and **Microsoft Azure** environments.

> 📌 **Important**
>
> Network connectivity issues are commonly caused by a missing firewall rule, incorrect routing, or an application listening only on the localhost interface.

---

# AWS Approach

## Troubleshooting Checklist

Verify the following components **in the order listed**:

| Step   | Component                       | Verification                                                                                                               |
| ------ | ------------------------------- | -------------------------------------------------------------------------------------------------------------------------- |
| **1**  | 🔒 Security Group Inbound Rules | Check if **port 8080** is open for **0.0.0.0/0** or the required CIDR range.                                               |
| **2**  | 🖥️ Instance State              | Ensure the EC2 instance is **Running** (not Stopped or Terminated).                                                        |
| **3**  | ✅ EC2 Status Checks             | Confirm both **System Status Check** and **Instance Status Check** show **2/2 Passed**.                                    |
| **4**  | 🌐 Public IP / Elastic IP       | Verify that the instance has a Public IP or Elastic IP assigned.                                                           |
| **5**  | 🛣️ Subnet                      | Ensure the instance resides in a **Public Subnet** associated with a route table that points to an Internet Gateway (IGW). |
| **6**  | 📍 Route Table                  | Verify the subnet route table contains **0.0.0.0/0 → igw-xxxxxxxx**.                                                       |
| **7**  | 🌍 Internet Gateway             | Confirm an Internet Gateway is attached to the VPC.                                                                        |
| **8**  | 🚧 Network ACL                  | Verify that the subnet-level Network ACL (NACL) does not block **port 8080**.                                              |
| **9**  | 🛡️ OS-Level Firewall           | Check **iptables** or **firewalld** inside the EC2 instance.                                                               |
| **10** | 🚀 Application Status           | Confirm the application is listening on **0.0.0.0:8080** and **not** **127.0.0.1:8080**.                                   |

---

## Fix Security Group

```bash
aws ec2 authorize-security-group-ingress \
--group-id sg-0abc123 \
--protocol tcp \
--port 8080 \
--cidr 0.0.0.0/0
```

---

## Verify Application Is Listening on the Correct Interface

```bash
ss -tlnp | grep 8080
```

Expected output:

```text
0.0.0.0:8080
```

> ⚠️ **Warning**
>
> The application **must** listen on **0.0.0.0:8080**.
>
> If it is listening on **127.0.0.1:8080**, external users will not be able to connect.

---

## Test Connectivity from an External Host

```bash
curl -v http://<public-ip>:8080/health
```

---

# Root Cause Analysis (RCA)

## Most Common Root Cause

> ❌ Missing Security Group inbound rule for **port 8080** **OR**
>
> ❌ The application is bound to **127.0.0.1 (localhost)** instead of **0.0.0.0**.

---

## Second Most Common Root Cause

> ❌ The subnet has **no route to the Internet Gateway (IGW)**.

---

## Resolution

* ✅ Add a Security Group inbound rule allowing **TCP 8080**.
* ✅ Restart the application and configure it to bind to **0.0.0.0** instead of **127.0.0.1**.

# Azure Approach

Perform the following checks when troubleshooting the equivalent scenario in Microsoft Azure.

## Verification Checklist

* 🔒 **NSG (Network Security Group):**

  * Add an inbound rule.
  * Port: **8080**
  * Source: **Any**
  * Priority: **100**

* 🌐 **Azure VM Public IP**

  * Assign a Public IP to the virtual machine.
  * Alternatively, use an **Azure Load Balancer** with a health probe configured on **port 8080**.

* 🛡️ **Azure Firewall**

  * If Azure Firewall is enabled within the Virtual Network (VNet), create the required **Application Rule** or **Network Rule** to allow traffic on **port 8080**.

* 💻 **Operating System Firewall**

  * Allow TCP port **8080**.

```bash
sudo ufw allow 8080/tcp
```

---

## Create an Azure NSG Rule

```bash
az network nsg rule create \
--resource-group prod-rg \
--nsg-name prod-nsg \
--name Allow-8080 \
--priority 100 \
--protocol Tcp \
--destination-port-range 8080 \
--access Allow
```

# Best Practice

> 💡 **Tip**
>
> Always verify **all five networking layers** before concluding that the application is unavailable.

1. 🔒 Security Group (AWS) / Network Security Group (Azure)
2. 🛣️ Route Table
3. 🚧 Network ACL (AWS)
4. 🌍 Internet Gateway (AWS)
5. 🛡️ Operating System Firewall

> 📌 Missing **any one** of these layers can block inbound traffic, even if every other component is correctly configured.
