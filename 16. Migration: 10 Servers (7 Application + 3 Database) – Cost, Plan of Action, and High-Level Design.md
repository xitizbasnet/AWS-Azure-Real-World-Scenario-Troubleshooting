# Migration: 10 Servers (7 Application + 3 Database) – Cost, Plan of Action, and High-Level Design

> **Scenario:** Migrate 10 on-premises servers:
>
> * **7 Application Servers**
> * **3 Database Servers**
> * **1 TB HDD storage per server**
>
> Objective:
>
> * Build a migration strategy.
> * Provide cost estimation.
> * Define Plan of Action (PoA).
> * Create High-Level Design (HLD).

---

# Overview

A successful cloud migration requires three major activities:

1. 🔍 **Assessment**

   * Discover servers.
   * Identify dependencies.
   * Analyze compatibility.

2. 🚀 **Migration**

   * Replicate workloads.
   * Migrate applications.
   * Move databases.

3. ✅ **Optimization**

   * Improve architecture.
   * Reduce cost.
   * Enable monitoring and security.

# Plan of Action (PoA)

# Phase 1: Discovery & Assessment (Week 1–2)

## Activities

* Install AWS MGN agent on all 10 servers.

* Run AWS Application Discovery Service.

* Document:

  * Operating system versions.
  * Application dependencies.
  * Open ports.
  * Database engines.
  * Data volume.

* Run AWS SCT for database compatibility testing.

* Identify licensing requirements:

  * Windows Server CALs.
  * Oracle licensing.
  * Third-party software licensing.

---

# Phase 2: Foundation Setup (Week 2–3)

## Activities

Create AWS foundation:

* Create VPC.

* Create public/private/database subnets across two Availability Zones.

* Setup:

  * Direct Connect OR
  * Site-to-Site VPN

* Create target:

  * EC2 instances.
  * RDS databases.

Configure:

* Security Groups.
* IAM Roles.
* S3 buckets.
* CloudWatch dashboards.
* GuardDuty.
* AWS Config.

---

# Phase 3: Non-Production Migration (Week 3–4)

## Activities

* Start MGN replication for two application servers.

* Start DMS full load migration for one development database.

* Test application functionality in AWS.

* Validate:

  * Performance.
  * Connectivity.
  * Dependencies.

* Fix identified issues.

* Perform User Acceptance Testing (UAT).

---

# Phase 4: Production Application Migration (Week 5–6)

## Activities

* Start MGN replication for all seven application servers.
* Monitor replication.
* Wait for:

```text
Replication Lag < 1 second
```

---

## Cutover Process

Schedule:

```text
Maintenance Window:
Low Traffic Period
Example: Sunday 2 AM
```

Steps:

1. Launch test instances in AWS.
2. Validate application health.
3. Update DNS:

```text
Route53
    │
    ▼
AWS Application Load Balancer
```

4. Monitor production for 24 hours.

---

# Phase 5: Database Migration (Week 6–7)

## Activities

* Perform DMS full-load migration for all databases.
* Enable CDC (Change Data Capture).
* Maintain real-time synchronization.

Migration flow:

```text
Application (AWS)
        │
        ▼
On-Prem Database
        │
        ▼
DMS CDC Replication
        │
        ▼
AWS RDS
```

---

## Database Cutover

Steps:

1. Validate data consistency.
2. Update application connection strings.
3. Point applications to RDS.
4. Verify functionality.
5. Stop DMS CDC replication.

---

# Phase 6: Post-Migration & Decommission (Week 8)

## Activities

Monitor for two weeks:

* CloudWatch metrics.
* Application logs.
* Performance metrics.

Additional actions:

* Compare performance against baseline.
* Enable automated backups.
* Enable RDS Multi-AZ.
* Decommission on-premises servers after 30-day parallel operation.
* Purchase Reserved Instances for cost optimization.

# Root Cause Analysis (RCA)

## Common Migration Failures

### Database Migration Without Compatibility Testing

Problem:

> Database types, schemas, or features may not be compatible.

Impact:

* Migration failures.
* Application errors.
* Data issues.

Solution:

* Run AWS SCT assessment before migration.

---

### Application Migration Without Dependency Mapping

Problem:

> Missing dependencies break application functionality.

Examples:

* External integrations.
* Required ports.
* Shared storage.
* Configuration files.

Solution:

* Document dependencies during discovery.

---

# Migration Best Practice

> 📌 Always run parallel operations (old + new environment) for a minimum of two weeks before decommissioning on-premises infrastructure.

Benefits:

* Safer rollback.
* Production validation.
* Reduced migration risk.
