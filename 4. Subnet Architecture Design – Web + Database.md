# Subnet Architecture Design – Web + Database

> **Scenario:** Web servers must be **internet-accessible**, while databases must **never** be publicly accessible.

---

# Overview

A secure and highly available network architecture separates workloads into multiple tiers. In a standard **3-tier architecture**, only the **presentation layer** is exposed to the internet, while the **application** and **database** tiers remain protected within private or isolated subnets.

This guide describes the recommended subnet architecture for **AWS** and its equivalent implementation in **Microsoft Azure**.

> 📌 **Important**
>
> Databases should **never** be deployed in public subnets. Restrict database access to the application layer using private networking and security group rules.

## Architecture Notes

* 🌐 **Application Load Balancer (ALB)** is deployed in **Public Subnets**.
* 🚀 ALB forwards requests to **EC2 Web/Application Servers** in **Private Subnets**.
* 🔒 The **Database Security Group** allows inbound traffic **only** from the **Application Server Security Group**.
* 🚫 Do **not** allow database access from **0.0.0.0/0**.
* 🛡️ Database subnets must **not** have any internet route, including routes through a NAT Gateway.

---

# Root Cause Analysis (RCA)

## Common Anti-Pattern

> ❌ Deploying the database in a **Public Subnet** while relying only on Security Group restrictions.

Although inbound rules may be restrictive, the database remains internet-routable and is exposed to unnecessary IP-based attack attempts.

---

## Resolution

* ✅ Create a dedicated subnet for each application tier.
* ✅ Place databases in **isolated/private subnets**.
* ✅ Configure Security Group rules using **Security Group references** instead of CIDR ranges whenever possible.
* ✅ Ensure the database subnet has **no route to the Internet Gateway or NAT Gateway**.

# Azure Approach

Implement an equivalent layered architecture using Azure networking services.

## Public Subnet

Deploy the following internet-facing components:

* 🌐 Azure Load Balancer
* 🛡️ Azure Application Gateway with Web Application Firewall (WAF)

---

## Private Subnet

Deploy application virtual machines with internal-only communication.

* 🖥️ Azure Virtual Machines
* 🛣️ User-Defined Routes (UDRs) to direct outbound traffic through Azure Firewall when required

---

## Database Subnet

Deploy managed database services using private connectivity.

Supported options include:

* Azure SQL Managed Instance
* Azure Database for PostgreSQL – Flexible Server

---

## Private Endpoint

Configure a **Private Endpoint** for Azure Platform-as-a-Service (PaaS) databases.

Benefits include:

* 🔒 Database accessible only through a private IP address
* 🌐 No public endpoint exposure
* 🛡️ Traffic remains within the Virtual Network (VNet)

---

# Best Practice

> 💡 **Tip**
>
> Enable **VPC Flow Logs** to monitor and audit network traffic between subnets.

Recommended workflow:

1. Enable **VPC Flow Logs**.
2. Store logs in **Amazon S3**.
3. Query traffic using **Amazon Athena**.
4. Use the results for:

   * Security investigations
   * Compliance audits
   * Network troubleshooting
   * Traffic analysis

> 📌 **Security Recommendation**
>
> Apply the principle of **least privilege** by allowing database access **only** from the application tier using Security Group references rather than IP address ranges.
