# Git Branch Conflict – Two Developers Pushed to the Same Branch

 
> **Scenario:** Two developers pushed code changes to the same Git branch at the same time, resulting in a merge conflict.

---

# Overview

A Git merge conflict occurs when Git cannot automatically combine changes from multiple developers.

The most common causes are:

* Multiple developers modifying the same file.
* Direct pushes to shared branches.
* Large commits with overlapping changes.
* Lack of Pull Request review workflows.

The correct resolution process is to:

1. Fetch the latest remote changes.
2. Review conflicting files.
3. Resolve conflicts manually.
4. Commit the resolved changes.
5. Push the updated branch.

> 📌 **Important**
>
> Never overwrite remote changes blindly. Always review conflicts and understand both versions of the code before merging.

---

# AWS Approach

> **Note:** AWS services such as CodeCommit or GitHub repositories use standard Git conflict resolution workflows. The Git commands below apply regardless of the hosting platform.

---

# Git Conflict Resolution Steps

Perform the following steps:

| Step  | Action                       | Description                                                    |
| ----- | ---------------------------- | -------------------------------------------------------------- |
| **1** | 📥 Fetch Latest Remote State | Download remote changes without automatically merging.         |
| **2** | 🌿 Create Temporary Branch   | Preserve the current working state before resolving conflicts. |
| **3** | 🔄 Rebase or Merge Changes   | Combine remote updates with local changes.                     |
| **4** | ✏️ Resolve Conflicts         | Open conflicted files and manually choose the correct changes. |
| **5** | ✅ Stage and Commit           | Mark conflicts as resolved and complete the merge.             |
| **6** | 🚀 Push Resolved Branch      | Upload the final resolved branch to the remote repository.     |

# Example Scenario

## Situation

```text
Developer A
    │
    ├── Modified app.js
    │
    ▼
feature/payment branch


Developer B
    │
    ├── Modified app.js
    │
    ▼
feature/payment branch
```

Both developers modified the same file, causing a conflict.

---

# Option 1: Merge Resolution

## Step 1: Fetch Remote Changes

```bash id="m6x9qa"
git fetch origin
```

---

## Step 2: Check Conflict Status

```bash id="h3p7vz"
git status
```

This shows which files contain conflicts.

---

## Step 3: Merge Remote Branch

```bash id="r8k4mw"
git merge origin/feature/payment
```

---

## Conflict Markers

Git adds conflict markers inside files:

```text id="t5n2yc"
<<<<<<< HEAD
your code here
=======
their code here
>>>>>>> origin/feature/payment
```

Meaning:

| Marker                           | Meaning                    |
| -------------------------------- | -------------------------- |
| `<<<<<<< HEAD`                   | Your local changes         |
| `=======`                        | Separation between changes |
| `>>>>>>> origin/feature/payment` | Remote branch changes      |

---

## Step 4: Resolve and Commit

After editing the conflicted file:

```bash id="v9q3fs"
git add app.js

git commit -m "resolve: merge conflict in payment module"
```

---

## Step 5: Push Changes

```bash id="b4n8kp"
git push origin feature/payment
```

---

# Option 2: Rebase Resolution

Rebase creates a cleaner commit history by replaying local changes on top of the latest remote branch.

---

## Step 1: Fetch Latest Changes

```bash id="z7m5hd"
git fetch origin
```

---

## Step 2: Start Rebase

```bash id="w2c8nr"
git rebase origin/feature/payment
```

---

## Step 3: Resolve Conflicts

After fixing conflicted files:

```bash id="p6y4kt"
git add app.js

git rebase --continue
```

---

## Step 4: Push Updated Branch

Because rebase rewrites commit history:

```bash id="q8v3mx"
git push origin feature/payment --force-with-lease
```

# Root Cause Analysis (RCA)

## Root Cause

> ❌ No branch protection rules were configured, allowing direct pushes to shared branches.

Additional causes:

* Two developers modified the same file simultaneously.
* No communication before editing shared components.
* Large pull requests increased conflict probability.

---

# Resolution and Prevention

## 1. Enable Branch Protection

Recommended rules:

* Require Pull Requests.
* Block direct pushes to protected branches.
* Require code reviews.
* Require successful CI builds.

---

## 2. Improve Team Communication

Before modifying shared files:

* Notify team members.
* Coordinate ownership of large changes.
* Avoid parallel edits on the same modules.

---

## 3. Create Smaller Pull Requests

Benefits:

* Easier reviews.
* Lower conflict probability.
* Faster merging.
* Easier rollback.

# Best Practice

> 💡 **Tip**
>
> Use `--force-with-lease` instead of `--force`.

Comparison:

| Command                       | Risk Level   | Description                                                |
| ----------------------------- | ------------ | ---------------------------------------------------------- |
| `git push --force`            | ⚠️ High Risk | Can overwrite another developer's changes.                 |
| `git push --force-with-lease` | ✅ Safer      | Fails if someone pushed new changes after your last fetch. |
